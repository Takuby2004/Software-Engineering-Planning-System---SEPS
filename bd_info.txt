CAPÍTULO 1. Arquitectura y Modelo Conceptual de Datos
================================================================================

1.1 Introducción al Ecosistema de Datos
La plataforma SEPS requiere una persistencia de datos altamente estructurada e 
íntegra. El sistema coordina tres modulos principales en el frontend (Wiki, Scrum 
Backlog y Diseñador de Base de Datos Virtual) y todos ellos deben persistirse en un 
único motor relacional robusto. La estrucura se diseña para que cada modificación 
sobre el proyecto registre de manera automática un log de auditoría histórica, 
garantizando la trazabilidad integral requerida en Ingeniería de Software.

1.2 Motor de Persistencia Relacional
El motor de base de datos seleccionado es **PostgreSQL (v15+)**, desplegado de 
forma altamente disponible en el servicio administrado **Google Cloud SQL**. Se optó 
por PostgreSQL debido a su estricto cumplimiento del estándar SQL, soporte avanzado 
para restricciones de integridad (CHECK, FOREIGN KEY con ON DELETE CASCADE), alta velocidad 
en consultas relacionales y soporte nativo para campos de texto largos donde se 
almacena información estructurada JSON (por ejemplo, los requerimientos funcionales 
y el esquema virtual de la base de datos).

1.3 Patrón de Acceso a Datos (ORM)
La comunicación entre la capa de negocio (Node.js con Express) y la base de datos 
se gestiona mediante **Drizzle ORM**. Este ORM de tipo TypeScript-first permite 
declarar el esquema físico como código TS puro (`src/db/schema.ts`), garantizando 
tipado estático estricto en tiempo de compilación y eliminando errores de desajuste 
de datos (impedance mismatch). La sincronización física se ejecuta mediante scripts 
de migración rápidos, asegurando que cualquier cambio en las entidades de Drizzle se 
refleje de inmediato en la base de datos PostgreSQL de Cloud SQL.

CAPÍTULO 2. Diagrama de Relaciones y Cardinalidades
================================================================================

2.1 Tabla de Relaciones Físicas
El diseño lógico de la base de datos consta de 6 entidades principales fuertemente 
acopladas mediante llaves foráneas para garantizar que la eliminación o modificación 
de un proyecto mantenga la coherencia del sistema sin dejar huérfanos.

Las cardinalidades definidas en el esquema físico son:
- **users (1) <----> (many) projects**: Un usuario registrado mediante Firebase Auth 
  puede poseer o administrar múltiples proyectos dentro del Hub, pero cada proyecto 
  pertenece obligatoriamente a un único usuario creador.
- **projects (1) <----> (many) scrum_iterations**: Un proyecto contiene múltiples 
  iteraciones o Sprints de Scrum definidos por el equipo para organizar el trabajo.
- **projects (1) <----> (many) scrum_tasks**: Un proyecto alberga muchas tareas 
  en su Backlog general.
- **scrum_iterations (1) <----> (many) scrum_tasks**: Un Sprint de Scrum agrupa 
  diversas tareas activas. Si una iteración se cancela o elimina, las tareas asociadas 
  permanecen en el backlog (la relación se establece a NULL en la llave foránea).
- **projects (1) <----> (many) test_cases**: Un proyecto de software tiene asociada 
  una suite de pruebas de caja negra e integración para validar los requisitos.
- **projects (1) <----> (many) project_changes**: Un proyecto contiene el registro 
  completo de auditoría con el historial de deltas de sus campos editados.

2.2 Integridad Referencial y Acciones en Cascada
Para asegurar que no existan tuplas huérfanas al eliminar registros lógicos, 
se han configurado las siguientes reglas de integridad referencial a nivel SGBD:
- `scrum_iterations.project_id` ON DELETE CASCADE: Si un proyecto se elimina, todos 
  sus sprints asociados se borran automáticamente de la base de datos física.
- `scrum_tasks.project_id` ON DELETE CASCADE: Al eliminar el proyecto, se eliminan 
  todas sus tareas asociadas para liberar almacenamiento.
- `scrum_tasks.iteration_id` ON DELETE SET NULL: Si un sprint se elimina de la base de 
  datos, las tareas asignadas a este sprint no se eliminan; en su lugar, vuelven al 
  Backlog general sin sprint asignado (`iteration_id` se vuelve NULL).
- `test_cases.project_id` ON DELETE CASCADE: Al remover un proyecto, se borran de 
  forma recursiva todos sus casos de prueba asociados.
- `project_changes.project_id` ON DELETE CASCADE: Al eliminar un proyecto, su bitácora 
  histórica de cambios de trazabilidad se elimina en cascada inmediatamente.

Esta tabla gestiona las cuentas de los usuarios autenticados en el sistema, vinculándose

CAPÍTULO 4. Script DDL de Creación Física (PostgreSQL)
================================================================================

El siguiente script contiene la especificación física DDL compatible con el dialecto 
de PostgreSQL para la creación de las tablas relacionales y sus respectivas restricciones.

```sql
-- SCRIPT DE CREACION FISICA - MOTORES POSTGRESQL (CLOUD SQL)
-- SEPS: Software Engineering Planning System

-- 1. Creación de Tabla de Usuarios
CREATE TABLE "users" (
    "id" SERIAL PRIMARY KEY,
    "uid" TEXT NOT NULL UNIQUE,
    "email" TEXT NOT NULL,
    "created_at" TIMESTAMP DEFAULT NOW()
);

-- 2. Creación de Tabla de Proyectos (Foco de documentación)
CREATE TABLE "projects" (
    "id" SERIAL PRIMARY KEY,
    "user_id" INTEGER NOT NULL,
    "name" TEXT NOT NULL,
    "description" TEXT,
    "organization" TEXT,
    "problem_context" TEXT,
    "org_description" TEXT,
    "identified_need" TEXT,
    "current_situation" TEXT,
    "main_problem" TEXT,
    "general_objective" TEXT,
    "specific_objectives" TEXT,
    "functional_requirements" TEXT,
    "non_functional_requirements" TEXT,
    "scope_limitations" TEXT,
    "wiki_notes" TEXT,
    "github_repos" TEXT,
    "architecture_type" TEXT,
    "architecture_description" TEXT,
    "languages_used" TEXT,
    "frameworks_used" TEXT,
    "databases_used" TEXT,
    "virtual_database_design" TEXT,
    "conclusions" TEXT,
    "recommendations" TEXT,
    "future_improvements" TEXT,
    "created_at" TIMESTAMP DEFAULT NOW(),
    "updated_at" TIMESTAMP DEFAULT NOW(),
    CONSTRAINT "fk_projects_users" FOREIGN KEY ("user_id") REFERENCES "users"("id") ON DELETE RESTRICT
);

-- 3. Creación de Tabla de Iteraciones Scrum (Sprints)
CREATE TABLE "scrum_iterations" (
    "id" SERIAL PRIMARY KEY,
    "project_id" INTEGER NOT NULL,
    "name" TEXT NOT NULL,
    "goal" TEXT,
    "start_date" TEXT,
    "end_date" TEXT,
    "status" TEXT DEFAULT 'Planning',
    "created_at" TIMESTAMP DEFAULT NOW(),
    CONSTRAINT "fk_iterations_projects" FOREIGN KEY ("project_id") REFERENCES "projects"("id") ON DELETE CASCADE
);

-- 4. Creación de Tabla de Tareas Scrum (Backlog Kanban)
CREATE TABLE "scrum_tasks" (
    "id" SERIAL PRIMARY KEY,
    "project_id" INTEGER NOT NULL,
    "iteration_id" INTEGER,
    "title" TEXT NOT NULL,
    "description" TEXT,
    "type" TEXT DEFAULT 'Feature',
    "priority" TEXT DEFAULT 'Medium',
    "status" TEXT DEFAULT 'To Do',
    "assigned_to" TEXT,
    "rf_code" TEXT,
    "created_at" TIMESTAMP DEFAULT NOW(),
    CONSTRAINT "fk_tasks_projects" FOREIGN KEY ("project_id") REFERENCES "projects"("id") ON DELETE CASCADE,
    CONSTRAINT "fk_tasks_iterations" FOREIGN KEY ("iteration_id") REFERENCES "scrum_iterations"("id") ON DELETE SET NULL
);

-- 5. Creación de Tabla de Casos de Prueba (QA Suite)
CREATE TABLE "test_cases" (
    "id" SERIAL PRIMARY KEY,
    "project_id" INTEGER NOT NULL,
    "code" TEXT NOT NULL,
    "name" TEXT NOT NULL,
    "description" TEXT,
    "preconditions" TEXT,
    "steps" TEXT,
    "expected_result" TEXT,
    "actual_result" TEXT,
    "status" TEXT DEFAULT 'Pending',
    "rf_code" TEXT,
    "evidence_notes" TEXT,
    "created_at" TIMESTAMP DEFAULT NOW(),
    CONSTRAINT "fk_testcases_projects" FOREIGN KEY ("project_id") REFERENCES "projects"("id") ON DELETE CASCADE
);

-- 6. Creación de Tabla de Auditoría Histórica (Project Changes)
CREATE TABLE "project_changes" (
    "id" SERIAL PRIMARY KEY,
    "project_id" INTEGER NOT NULL,
    "field_name" TEXT NOT NULL,
    "old_value" TEXT,
    "new_value" TEXT,
    "changed_at" TIMESTAMP DEFAULT NOW(),
    CONSTRAINT "fk_changes_projects" FOREIGN KEY ("project_id") REFERENCES "projects"("id") ON DELETE CASCADE
);
```

CAPÍTULO 5. Optimización, Seguridad e Índices
================================================================================

5.1 Índices sugeridos para consultas frecuentes
Para garantizar tiempos de respuesta ultra-bajos e impedir degradación en el rendimiento 
del servidor Express (cumpliendo con la latencia inferior a 2 segundos declarada), 
se proponen los siguientes índices no agrupados:

```sql
-- Optimiza la búsqueda de proyectos pertenecientes a un usuario en el Dashboard
CREATE INDEX "idx_projects_user_id" ON "projects" ("user_id");

-- Agiliza la carga de las tareas filtradas por Sprint activo en el Kanban
CREATE INDEX "idx_scrum_tasks_iteration" ON "scrum_tasks" ("iteration_id");

-- Optimiza las consultas de auditoría que recuperan los últimos 5 cambios del proyeto ordenados por fecha
CREATE INDEX "idx_project_changes_project_date" ON "project_changes" ("project_id", "changed_at" DESC);

-- Agiliza la correspondencia de requerimientos funcionales mapeados a pruebas y tareas
CREATE INDEX "idx_scrum_tasks_rf" ON "scrum_tasks" ("rf_code");
CREATE INDEX "idx_test_cases_rf" ON "test_cases" ("rf_code");
```

5.2 Seguridad de datos e Integridad de Usuarios
- **Tokens y Secretos**: No se almacenan contraseñas directamente en nuestra base 
  de datos PostgreSQL. La seguridad de autenticación está delegada en Firebase Auth. 
  La tabla `users` contiene únicamente el identificador UUID seguro proporcionado 
  por Google Firebase. De este modo, si ocurre un acceso no autorizado a nivel SGBD, 
  no hay riesgo de exposición de credenciales hash.
- **Consultas con Parámetros**: Todas las llamadas SQL se realizan mediante el 
  motor de consultas parametrizadas de Drizzle ORM, previniendo ataques de inyección SQL (SQLi).

CAPÍTULO 6. Conclusiones y Recomendaciones
================================================================================

6.1 Conclusiones
  1. El esquema relacional diseñado de 6 tablas resuelve de manera balanceada 
     la persistencia del Hub de Proyectos sin redundancia destructiva.
  2. Las restricciones físicas de ON DELETE CASCADE e integridad referencial 
     impiden la proliferación de tuplas huérfanas en las tablas dependientes.
  3. El uso de campos TEXT mapeados a arreglos JSON simplifica la persistencia de 
     datos jerárquicos variables como los requerimientos funcionales, optimizando 
     la flexibilidad frente a esquemas extremadamente complejos.

6.2 Recomendaciones
  1. Mantener los índices idx_project_changes_project_date actualisados para evitar 
     que el crecimiento desmedido de la bitácora de cambios ralentice la Wiki.
  2. Implementar un cronjob nocturno para purgar cambios antiguos en `project_changes` 
     que superen los límites de los últimos 5 deltas por proyecto si se desea ahorrar 
     almacenamiento físico.

6.3 Mejoras Futuras en el Esquema de Datos
  1. Soporte para múltiples editores colaborativos añadiendo un campo `user_id` en 
     la tabla `project_changes` para identificar quién realizó el cambio.
  2. Migración del campo de base de datos virtual de TEXT a tipo JSONB para habilitar 
     búsquedas directas dentro del diccionario de datos desde PostgreSQL.
