CAPÍTULO 1. Descripción del proyecto
================================================================================

1.1 Introducción
- Contexto del problema: El desarolo moderno de sistemas informáticos requiere una 
alta sincronización entre los requerimientos declarados en la etapa de planificación 
y la estrucura de datos implementada físicamente en el motor relacional. Sin embargo, 
durante el ciclo de vida del software, es muy común que se pierda la trazabilidad de 
las modificaciones, resultando en scripts de bases de datos desactualisados y sprints 
con requerimientos desalineados.
- Descripción de la organización o entorno: El entorno donde se aplica esta solución 
son los equipos de desarrollo ágil y cursos de Ingeniería de Software donde se 
diseñan soluciones informáticas de nivel empresarial. Estos equipos demandan un 
sistema centralisado donde registrar la información del proyecto y sus cambios en 
tiempo real.
- Necesidad identificada: Se identificó la necesidad critica de automatizar la 
trazabilidad de las especificaciones de un proyeto. Cuando un gestor o arquitecto 
modifica un campo descriptivo, los desarrolladores deben conocer con precisión qué 
cambió, cuándo y cuál era el valor anterior para evitar costosos errores de diseño.

1.2 Problema
- Situación actual: Los equipos suelen utilizar herramientas desconectadas: planillas 
de cálculo para la wiki, gestores de tareas externos para el backlog Scrum, y software 
independiente de modelado para el DER. Esto fragmenta la información y oculta los 
cambios críticos realizados a los campos del proyeto.
- Problema principal: ¿Cómo integrar de manera coherente y trazable el modelado 
físico de bases de datos, la gestión ágil con Scrum, y la auditoría de cambios en 
una plataforma web unificada para optimizar el desarolo de software?

1.3 Objetivos
- Objetivo General: Desarrollar una solución de software web funcional y documentada 
denominada SEPS (Software Engineering Planning System) aplicando principios de 
Ingeniería de Software para centralizar el modelado relacional, el control de backlog 
de sprints, y registrar el historial de los últimos 5 cambios de los campos del proyeto.
- Objetivos Específicos:
  1. Diseñar e implementar un componente persistente de historial de cambios para 
     registrar modificaciones en los campos clave del proyecto en tiempo real.
  2. Implementar un módulo interactivo para diseñar la base de datos de manera virtual 
     con capacidades de autogeneración de scripts SQL DDL para PostgreSQL.
  3. Desarrollar un panel Kanban interactivo para planificar iteraciones con el framework Scrum.
  4. Validar la consistencia y escalabilidad de la arquitectura en capas implementada.

1.4 Alcance
- Requisitos funcionales: Creación y edición de proyectos, tracking en tiempo real de 
campos editados, creación de historias de usuario y tareas de Scrum, estructuración 
de tablas relacionales y exportación de DDL, autenticación segura con Firebase.
- Requisitos no funcionales: Interfaz web responsiva con Tailwind CSS v4, base de datos 
relacional PostgreSQL en la nube (Cloud SQL), latencia de carga inferior a 2 segundos.
- Funcionalidades incluidas: Historial interactivo de cambios (últimos 5 cambios por 
proyecto), tablero Kanban de sprints, diseñador visual de esquema relacional físico.
- Limitaciones del proyeto: El sistema no genera scripts DDL específicos para motores 
no relacionales como MongoDB, enfocándose exclusivamente en estándares SQL (PostgreSQL).

CAPÍTULO II. Marco Teórico
================================================================================

A. Marco metodológico

2.1 Ingeniería de Software
- Conceptos: Disciplina de la ingeniería que se ocupa de todos los aspectos de la 
producción de software, desde las etapas iniciales de la especificación del sistema 
hasta el mantenimiento posterior a su puesta en marcha.
- Principios: Rigor y formalidad, separación de intereses, modularidad, abstracción, 
anticipación al cambio, generalidad e incremento.
- Ciclo de vida: Modelos estructurados para planificar, diseñar, construir, probar 
y desplegar software sistemáticamente.

2.2 Metodología de Desarolo
- Ágil: Enfoque iterativo para el desarrollo de software que promueve la colaboración, 
la adaptabilidad al cambio y la entrega continua de valor. Se basa en el Manifiesto Ágil.

2.3 Framework Scrum
- Roles: Product Owner (dueño del producto), Scrum Master (facilitador de procesos), 
y Developers (equipo de desarolo).
- Eventos: Sprint Planning, Daily Scrum, Sprint Review, y Sprint Retrospective.
- Artefactos: Product Backlog, Sprint Backlog, e Incremento de software.

2.4 UML (Unified Modeling Language)
- Diagramas estructurales: Muestran la organización estática del sistema. El más 
representativo es el Diagrama de Clases.
- Diagramas de comportamiento: Describen la dinámica y flujo de acciones. Incluyen 
el Diagrama de Secuencia, Diagrama de Actividad y Casos de Uso.

2.5 Arquitetura de Software
- MVC (Modelo-Vista-Controlador): Patrón de arquitetura que separa los datos de la 
aplicación, la interfaz de usuario y la lógica de control.
- Arquitectura en capas: Organización jerárquica donde cada nivel ofrece servicios a la 
capa superior (Presentación -> Negocio -> Datos). Esta es la que rige SEPS.

2.6 Bases de Datos
- Sistemas gestores de bases de datos (SGBD): Software para definir, crear, mantener 
y controlar el acceso a la base de datos (por ejemplo, PostgreSQL).
- Modelado de datos: Proceso de creación de un modelo para los datos que se van a 
almacenar (Conceptual, Lógico y Físico).
- Normalización: Proceso de organización de los datos para evitar la redundancia y 
garantizar la integridad (1FN, 2FN, 3FN).

2.7 Tecnologías Utilizadas
- Lenguajes: TypeScript (superconjunto de JavaScript con tipado estático).
- Frameworks: React para el frontend interactivo y Express para el backend server-side.
- Herramientas CASE: Drizzle ORM para modelado de datos físico y sincronización rápida.
- Gestor de base de datos: PostgreSQL alojado en Cloud SQL.

B. Marco Tecnológico o de innovación (Trazabilidad en Tiempo Real)
- Definición: La trazabilidad reactiva es una técnica que captura automáticamente 
cualquier cambio de estado en los metadatos de configuración de un software, 
registrando los deltas antes de su persistencia física.
- Características: Persistencia no destructiva, auditoría cronológica y renderizado 
instantáneo de diferencias en la interfaz de usuario.
- Fases de implementación: 1. Intercepción de peticiones en API, 2. Comparación 
de deltas, 3. Inserción asíncrona de bitácora, 4. Exposición mediante API GET.
- Ventajas: Mayor control de cambios, auditoría inmediata, reducción de malos 
entendidos sobre los requerimientos.
- Desventajas: Pequeño overhead en las transacciones de escritura de la base de datos.
- Usos y aplicaciones: Sistemas financieros, herramientas de planeación de software, 
gestores de configuración (CMS).

CAPÍTULO 3. Análisis de requerimientos
================================================================================

3.1 Descripción General del Sistema
SEPS es una plataforma interactiva full-stack diseñada para el control y 
documentación de proyectos de software. Integra en una sola interfaz la wiki descriptiva, 
el backlog ágil de iteraciones Scrum, el visualizador relacional físico de bases de datos 
y un componente de trazabilidad que audita de manera automática cualquier cambio 
en los datos del proyecto.

3.2 Actores del Sistema
- Gestor de Proyeto / Product Owner: Crea el proyecto y define los requerimientos.
- Desarrollador / Scrum Master: Implementa funcionalidades, gestiona tareas en el Kanban 
  y consulta el diseño de base de datos y scripts DDL.

3.3 Diagrama de Procesos de Negocio
[Inicio] -> [Crear Proyecto] -> [Modificar Requerimientos] 
                              |-> (Captura automática del cambio -> Guardar en Tabla project_changes)
                              v
                      [Planificar Sprint] -> [Diseñar Base de Datos Virtual] -> [Exportar DDL SQL] -> [Fin]

3.4 Requerimientos Funcionales
- RF01: El sistema debe permitir el registro e inicio de sesión seguro de usuarios.
- RF02: El sistema debe permitir crear, leer, actualizar y eliminar proyectos.
- RF03: El sistema debe interceptar cualquier edición a los campos descriptivos del proyecto y guardar el cambio.
- RF04: El sistema debe exponer un servicio para obtener los últimos 5 cambios realizados a un proyecto específico.
- RF05: El componente visual debe renderizar el valor anterior y el valor nuevo del campo modificado.
- RF06: El sistema debe permitir la creación de Sprints/Iteraciones Scrum.
- RF07: El sistema debe permitir agregar historias de usuario y tareas con estados (To Do, In Progress, Done).
- RF08: El sistema debe poseer un diseñador de bases de datos virtual para agregar tablas y columnas.
- RF09: El sistema debe generar scripts de creación física SQL compatibles con PostgreSQL de forma automática.
- RF10: El sistema debe contar con un botón para refrescar manualmente el historial de trazabilidad.

3.5 Requerimientos No Funcionales
- RNF01: El sistema debe soportar conexiones HTTPS seguras utilizando Firebase Auth.
- RNF02: La persistencia de datos debe ser duradera y gestionada a través de un motor PostgreSQL en la nube.
- RNF03: La latencia de actualización de los cambios del proyecto debe ser inferior a 1 segundo.
- RNF04: La interfaz del usuario debe adaptarse a pantallas móviles y de escritorio de forma fluida.
- RNF05: El sistema debe controlar accesos no autorizados a la API de cambios mediante middleware de autenticación.

3.6 Diagrama de Navegación
Dashboard Proyectos -> Seleccionar Proyecto -> Panel de Pestañas:
  - Información General + Historial de Cambios (Últimos 5 registros)
  - Gestión Scrum (Backlog y Kanban)
  - Diseño de Base de Datos (Virtual DB)
  - Repositorio Código (GitHub Sync)

3.7 Planificación de iteraciones con Scrum
El backlog de SEPS se divide en iteraciones cortas de una semana:
- Sprint 1: Autenticación de usuarios y estructura base.
- Sprint 2: Diseñador visual de base de datos relacional.
- Sprint 3: Gestión de tareas con Kanban interactivo.
- Sprint 4: Módulo de trazabilidad e historial de auditoría de cambios en tiempo real (Últimos 5 cambios).

3.8 Casos de Uso (Diagrama General de alto nivel)
                       +---------------------------------------+
                       |           SEPS - Sistema              |
                       |                                       |
    ( Product Owner )--|---> [ Gestionar Proyecto ]            |
                       |              |                        |
                       |              v                        |
                       |     [ Registrar Cambio de Campo ]     |
                       |                                       |
    ( Desarrollador )--|---> [ Consultar Historial Trazabilidad] |
                       |                                       |
                       |---> [ Gestionar Tablas DB Virtual ]   |
                       |                                       |
                       |---> [ Mover Tareas en Kanban ]        |

CAPÍTULO 4. Diseño del sistema
================================================================================

4.1 Arquitectura General
El sistema implementa una arquitetura en capas clásica:
- Capa de Presentación (Frontend): React SPA estructurada con Tailwind CSS v4 para un 
  diseño visual pulido y responsivo, consumiendo APIs REST de manera segura.
- Capa de Negocio (Backend): Node.js + Express que procesa las peticiones de cambio 
  de campos del proyecto, intercepta diferencias y ejecuta lógica de auditoría.
- Capa de Datos (Persistencia): Motor PostgreSQL consultado mediante Drizzle ORM, 
  garantizando integridad referencial y velocidad transaccional.

4.2 Diagrama de Clases
   +--------------------+             +------------------------+
   |      Project       |             |     ProjectChange      |
   +--------------------+             +------------------------+
   | - id: Integer      |1          * | - id: Integer          |
   | - name: String     |<------------| - projectId: Integer   |
   | - description: Text|             | - fieldName: String    |
   | - organization: Str|             | - oldValue: Text       |
   |                    |             | - newValue: Text       |
   +--------------------+             | - changedAt: Timestamp |
                                      +------------------------+

4.3 Diagrama de secuencia (General)
Usuario              Frontend (React)           Backend (Express)            Base de Datos (PG)
   |                        |                           |                             |
   |--- Modifica Campo ---->|                           |                             |
   |    (ej. Objetivo G.)   |--- PUT /api/projects/:id->|                             |
   |                        |                           |--- Buscar proyecto actual ->|
   |                        |                           |<-- Retorna proyecto viejo --|
   |                        |                           |--- Compara deltas y detecta |
   |                        |                           |    campos alterados --------|
   |                        |                           |--- Inserta log en cambs --->|
   |                        |                           |--- Actualiza proyecto ----->|
   |                        |                           |<-- Confirmación OK ---------|
   |                        |<-- Retorna 200 OK --------|                             |
   |<-- Renderiza cambios --|                           |                             |

4.4 Diagrama de Actividad (General)
[Modificación de Campo] -> [¿El nuevo valor es idéntico?] 
                              |---> SÍ ---> [Ignorar y continuar]
                              |---> NO ---> [¿Hay cambio reciente de este campo < 15 seg?]
                                               |---> SÍ ---> [Actualizar valor nuevo del registro]
                                               |---> NO ---> [Insertar nueva tupla en project_changes]

4.5 Diseño de base de datos

- Modelo Conceptual:
  Un PROYECTO contiene múltiples TAREAS de Scrum, múltiples ITERACIONES, múltiples 
  CASOS DE PRUEBA y múltiples CAMBIOS DE PROYECTO (project_changes) para fines de auditoría.

- Modelo Lógico (Esquema relacional simplificado):
  - projects (id PK, user_id FK, name, description, problem_context, objective, etc.)
  - project_changes (id PK, project_id FK, field_name, old_value, new_value, changed_at)
  - scrum_iterations (id PK, project_id FK, name, start_date, end_date)
  - scrum_tasks (id PK, project_id FK, iteration_id FK, title, status)

- Modelo Físico:
  CREATE TABLE "project_changes" (
	"id" serial PRIMARY KEY NOT NULL,
	"project_id" integer NOT NULL,
	"field_name" text NOT NULL,
	"old_value" text,
	"new_value" text,
	"changed_at" timestamp DEFAULT now()
  );
  ALTER TABLE "project_changes" ADD CONSTRAINT "project_changes_project_id_projects_id_fk" 
  FOREIGN KEY ("project_id") REFERENCES "projects"("id") ON DELETE cascade;

- Diccionario de Datos (Tabla project_changes):
  - id: SERIAL (Clave Primaria, identificador único de cambio).
  - project_id: INTEGER (Clave Foránea, vinculada a projects.id).
  - field_name: TEXT (Nombre técnico del campo modificado).
  - old_value: TEXT (Contenido del campo antes de la modificación).
  - new_value: TEXT (Contenido del campo posterior a la modificación).
  - changed_at: TIMESTAMP (Fecha y hora exacta del registro del cambio).

CAPÍTULO 5. Implementación
================================================================================

5.1 Tecnologías Utilizadas
- Lenguaje: TypeScript para asegurar robustez en el tipado de modelos de datos.
- Framework: React (v19) para el renderizado eficiente de componentes reactivos.
- Estilos: Tailwind CSS v4 brindando animaciones fluidas y aspecto visual moderno.
- Middleware de Servidor: Express con NodeJS manejando ruteo y autenticación por tokens.
- ORM: Drizzle ORM para consultas rápidas sin overhead innecesario.

5.2 Desarolo de Funcionalidades

- Módulo 1: Gestión de Trazabilidad e Historial de Cambios
  Se implementó un middleware automático en la ruta de actualización de proyectos. 
  Compara el valor existente en la BD con el enviado por el cliente. Si hay cambios, 
  los registra en la tabla `project_changes`. Para evitar redundancia excesiva al 
  escribir letra por letra, se implementó un mecanismo de ventana de 15 segundos: si se 
  modifica el mismo campo en menos de 15 segundos, se actualiza el último registro en 
  lugar de insertar una nueva fila. El componente `ProjectChangeHistory` consume la API 
  para renderizar los últimos 5 cambios en formato colapsable.

- Módulo 2: Backlog de Sprints y Scrum Iterations
  Permite estructurar las iteraciones asignando fechas específicas y organizando tareas 
  mediante un tablero Kanban totalmente interactivo donde arrastrar las tarjetas actualiza 
  el estado físico en la base de datos de inmediato.

- Módulo 3: Visualizador y Generador DDL de BD
  Un modelador interactivo integrado de tablas y campos virtuales que genera scripts 
  físicos DDL estructurados listos para ser corridos en PostgreSQL.

5.3 Entregables
- Sistema web completamente funcional montado en Cloud Run.
- Casos de prueba ejecutados satisfactoriamente cubriendo flujos de edición.
- Historial de cambios mostrando con precisión los deltas de textos de wiki.

CAPÍTULO 6. Conclusiones y recomendaciones
================================================================================

6.1 Conclusiones
  1. El sistema SEPS cubre de manera sobresaliente el objetivo de integrar 
     planificación ágil y modelado físico de bases de datos.
  2. El modulo de historial de cambios simple implementado cumple con la cuota de 
     ofrecer visibilidad inmediata de los últimos 5 cambios de variables críticas 
     del proyecto para el equipo de desarrollo.
  3. La arquitectura en capas elegida evitó el acoplamiento y facilitó añadir la 
     trazabilidad de datos sin afectar el desempeño general de la interfaz.

6.2 Recomendaciones
  1. Utilizar siempre el botón de actualización asíncrona del historial para sincronizar 
     cambios realizados de forma colaborativa por otros usuarios.
  2. Realizar migraciones constantes del esquema físico de base de datos utilizando 
     las herramientas integradas de SEPS.

6.3 Mejoras Futuras
  1. Añadir una extensión para rastrear commits directamente desde repositorios de GitHub.
  2. Implementar notificaciones automáticas vía correo electrónico utilizando la 
     bitácora de trazabilidad de cambios cuando se alteren objetivos generales.
